# 1. Introduction

This report presents the Capstone II phase of **Modula POS**. Modula is a point-of-sale system intended for real business operation, not only for classroom demonstration. By the time this phase began, the project had already moved beyond initial design discussion and into a working prototype. That prototype was useful, but it also made several weaknesses visible. Capstone II therefore focuses on improving the system where the first prototype was weak, while continuing the implementation toward a more credible staged product.

The most important issues exposed by the Capstone I prototype were not cosmetic ones. They were structural. The early version did not yet handle multi-tenant access cleanly, branch and tenant workspaces were not clearly separated, offline behavior was not defined strongly enough, and several cross-module rules were too loose for reliable implementation. Because of that, Capstone II is not mainly about adding many new screens or extra features. Its main role is to correct those structural problems and make the system more consistent across frontend, backend, and business specification.

This does not mean Capstone II only fixes old work. It also introduces several practical additions that were needed once the system was treated more seriously: stronger account onboarding, a clearer multi-tenant operating model, real KHQR payment handling, attendance evidence improvements, automatic inventory deduction, and preparation for subscription-aware billing. At the current stage, both frontend and backend are close to completion and have been deployed to an online staging environment. What remains is mostly refinement, integration validation, and final reporting. For that reason, this report documents not only what Modula POS is, but also how the project changed after the first prototype exposed what had to be rebuilt.

## 1.1 Presentation of Capstone

In the third year of our study program, the capstone project is intended to connect academic study with practical software development. It requires students to take ideas from software engineering, system analysis, database systems, project management, and related subjects, then apply them in one project rather than treating each subject separately. The result is expected to include not only design documents, but also implementation work, technical decisions, and formal reporting.

For Modula POS, this requirement is reflected in the full project lifecycle already visible in the work: problem analysis, requirement definition, architecture and data design, specification refinement, frontend and backend implementation, deployment preparation, and evaluation planning. In this project, the capstone is not treated only as a conceptual exercise. It is used to show whether the team can take a system from idea, to prototype, to a staged implementation that can be reasoned about and tested under more realistic operating conditions.

---

## 1.1.1 Objective of Capstone

The main objective of this capstone is to bring together the theoretical and practical knowledge acquired throughout the degree program and apply it to one coherent software system. In the case of Modula POS, that means taking a real operational problem, analyzing it, designing around it, and then implementing the result in a way that can still grow without collapsing under added complexity.

Another objective is to simulate the conditions of actual software work. This includes teamwork, communication across frontend and backend concerns, scope control, iterative refinement, and architectural decision-making. In this project, those aspects became especially important once the first prototype exposed flaws that could not be solved by surface-level UI changes alone.

Finally, the capstone must produce documentation that is academically defensible and still useful to the project itself. The report is expected to explain the problem context, the reasoning behind major design decisions, the current implementation state, and the remaining work. In other words, it should not simply describe a finished idea. It should document how the system evolved, what had to be corrected, and what has been achieved so far.

## 1.1.2 Duration of Capstone

The capstone spans approximately six months across two phases: **Capstone I** and **Capstone II**.

Capstone I was conducted from **1st October 2025** to **17th December 2025**.

Capstone II continues from the end of Capstone I and spans from **12th January 2026** to **22nd March 2026**.
